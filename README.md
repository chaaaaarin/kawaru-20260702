## 第1章 なぜ「AI社員」から「ループ」へ進化したのか

このチャンネルでは前回まで、**「AI社員を雇って」SEOオウンドメディアを完全自動化する**方法を解説してきた 🔶（自チャンネルの過去動画：Claude Code版「AI社員を8人雇って記事作成を完全自動化」／Kawaru版「Codex社員を大量に採用してオウンドメディアを自動運用」）。仕組みは、指揮官（Claude Code / Codex）＋担当エージェントが、キーワード選定→リサーチ→構成→執筆→品質チェック（95点以上）→WordPress公開→GA/GSC分析→リライトを回す、というもの。**ヒューマンインザループ**（最終公開判断は人間）で、スケジュールタスクにより朝5時・午後2時・夜10時に定期実行していた。

これは十分に強力だが、いま振り返ると **「時間で起動する自動化」に寄っていた**。ここに、Anthropicが2026年6月30日に公開した公式記事「Getting started with loops」の枠組みを重ねると、次に伸ばせる余地がはっきり見える ✅。同記事はループを **「エージェントが、決められた停止条件を満たすまで作業のサイクルを繰り返すこと」** と定義し、4つの軸（何をきっかけに始まる／何で止まる／どの機能を使う／どんなタスク向きか）で整理している。

つまり今回の主題は、**「AIに記事を書かせる」から「AIが改善し続けるループを設計する」への一段の進化**。前回との違いを一枚で言うと次のとおり。

| 観点 | 前回（AI社員シリーズ）🔶 | 今回（ループエンジニアリング）✅ |
|---|---|---|
| 考え方 | AI社員を「雇って」書かせる | ループを「設計」して自律的に回す |
| 起動 | スケジュール（時間）で定期実行 | 4種類のトリガー、特に**イベント駆動（プロアクティブ）** |
| 停止 | 品質95点で公開判断 | **停止条件を明示設計**（`/goal`・最大試行回数・検証役モデル） |
| 検証 | 人が最終チェック | **実行役と検証役を分ける**（Doer-Verifier） |
| 積み上げ | 分析を次記事に反映 | **共有メモリ＋北極星で複利**（回すほど賢くなる） |

- 進化の系譜：Prompt → Context → Harness → **Loop** Engineering ✅
- ループの定義（公式）：**停止条件を満たすまでサイクルを繰り返す** ✅
- 「すべてのタスクに複雑なループは不要。シンプルから始め、必要な場面だけパターンを選ぶ」も公式の重要メッセージ ✅

## 第2章 ループの4種類（Anthropic公式）をSEOに当てはめる

「Getting started with loops」は、ループを**4種類**に整理している ✅。SEO業務に対応づけると、前回どこまでやっていて、次に何を足せるかが見える。

**① ターンベースループ** — きっかけ：人がプロンプト送信／止まる：Claudeが「完了」or「要追加情報」と判断。短め・非定型のタスク向き。
→ SEOでは「この記事の構成を相談」「この段落を直して」など**都度の相談・下書き**。`SKILL.md` に検証手順（記事チェックリスト等）を書くと、Claude自身の自己検証が効く ✅。

**② ゴールベースループ（`/goal`）** — きっかけ：その場のプロンプト／止まる：ゴール達成 or 最大試行回数。検証可能な条件があるタスク向き。評価役モデルが達成を判定する ✅。
→ SEOでは「**品質スコア95点以上になるまで改稿**」「目標順位に到達するまで」。前回の「95点で公開」を、**停止条件つきのループ**として体系化できる。例：`ホームページのLighthouseを90点以上に。5回試してダメなら停止`。

**③ タイムベースループ（`/loop`・`/schedule`）** — きっかけ：指定した時間間隔／止まる：自分で止める or 作業完了。繰り返し作業・外部システム連携向き ✅。
→ **前回の「朝5時・午後2時・夜10時に起動」はまさにこれ**。`/loop` は手元PC、`/schedule` はクラウド常駐の「ルーティン」。毎朝の順位チェックや定期リライトが該当。

**④ プロアクティブループ** — きっかけ：イベント or スケジュール（人がリアルタイム関与しない）／止まる：個々のタスクはゴール達成、ルーティン自体は止めるまで継続。バグ報告・トリアージなど**内容が明確で繰り返す作業**向き ✅。「オートモード」「ダイナミックワークフロー」（研究プレビュー）を組み合わせる。
→ **ここが今回の新要素**。「順位が下がったら」「GSCにエラーが出たら」**イベントで自動起動**し、複数記事を並行でトリアージ→改稿→レビューまで進める。時間ではなく“できごと”に反応する。

公式のまとめ表（何を任せ、いつ使うか）：

| ループ | 何を任せるか | 使うタイミング | 主な機能 |
|---|---|---|---|
| ターンベース | 確認作業 | 探索中・検討中 | 検証用スキル |
| ゴールベース | 停止条件 | 「完了」の定義が明確 | `/goal` |
| タイムベース | きっかけ | 外部で定期的に発生 | `/loop`・`/schedule` |
| プロアクティブ | 指示そのもの | 繰り返し発生・内容が明確 | 上記全部＋ダイナミックワークフロー |

- 前回＝主に③。今回＝②＋④まで広げる、が進化のキモ
- どれも「シンプルから始め、必要な場面だけ複雑にする」が公式の基本姿勢 ✅

## 第3章 SEOループのワークフロー

単一の万能エージェントに全部やらせると各工程が凡庸になる。**専門特化エージェントの分業**が有効 🔶。動画の構成に沿って6つの役割に分ける。

| エージェント | 役割 |
|---|---|
| **Research Agent** | キーワード・SERP・競合を調査しブリーフ化 |
| **Writer Agent** | ブリーフをもとに執筆 |
| **Reviewer Agent** | 事実・E-E-A-T・読みやすさを点検 |
| **SEO Checker** | SEO/GEOスコア・内部リンク・構造化データを検査 |
| **Publisher** | CMS入稿・schema付与・整形 |
| **Monitor** | 順位・CTR・インデックスを監視し次ループを起こす |

このワークフローを強くするために、Anthropic公式の「human-agent teams」「Steering Claude Code」から3つの要点を足す ✅。

1. **実行役と検証役を分ける（Doer-Verifier）** — 起案するエージェントと、その成果を検証するエージェントを分けると、自己採点バイアスが減り品質が安定する ✅。SEOでは Writer と Reviewer/SEO Checker の分離がこれにあたる。
2. **共有メモリ（file-based knowledge base）に学びを残す** — 各ループは「何を見つけ何をしたか」を `LOG.md` 等に追記し、次回はそれを読んで続きから動く。**複数ループ（SEO・広告・プロダクト）が同じフォルダを読み書き**すると、広告ループが見つけたKWがSEOループに流れ込む——これが**複利（compounding）**。「公開の場で働く」＝エージェントが読める形で情報を残すのが前提 ✅。
3. **役割・北極星・段階的な信頼** — 「名簿」で誰が何を担うかを明文化し、野心的な長期ゴール（北極星）を共有。最初は人がレビューし、実績に応じて自律範囲を広げる ✅。

AIエージェント（Claude Code・Codex 等）の **スキル / サブエージェント / MCP / フック** が各エージェントとガードレールの実体になる。「絶対にしてはいけないこと」は指示ではなく**フック＋権限設定**で止めるのが公式の考え方 ✅。

## 第4章 実践例（4つのループ）

考え方を具体シーンに落とすと、次の4つが作れる。いずれも「トリガー → 作業 → ログ → 次のトリガー」の形。

**① 新規記事制作ループ（ゴールベース）** — Research → Writer → Reviewer → SEO Checker → Publisher を直列で回し、`/goal` で「品質スコア95点以上」「Reviewerの必須指摘が全解消」を停止条件にする。検証役モデルが達成を判定 ✅。

**② 既存記事改善ループ（プロアクティブ）** — Monitor が「順位が下がった記事」を**イベントとして検知して起動**。競合の更新点を調べて該当箇所を自動リライト、再検査して再公開。人が毎日順位表を眺める必要がなくなる。

**③ Search Console改善ループ（タイム／プロアクティブ）** — `/schedule` でGSCを定期チェック（＋下落は即トリガー）。低CTRページのタイトル書き換え、内部リンク補強、インデックスエラーのエスカレーションを提案・実行 🔶。2025年12月にGSCへ **AI-powered configuration** が追加された ✅。

**④ E-E-A-Tチェックループ（Doer-Verifier）** — Reviewer/SEO Checker を**検証役**に立て、Googleの品質評価ガイドライン観点（Experience・Expertise・Authoritativeness・Trust）で毎回チェック。2022年末にE-A-Tへ「Experience」が追加されE-E-A-Tに ✅。2026年版では**AI生成でも品質基準は同一**、AI Overviews評価の章が新設 ✅（E-E-A-Tは直接のランキング要因ではない点に注意 ✅）。

> 🔗 **実際にこの「ループでSEOを回す」考え方で運用されているAIエージェント特化メディアがこちら → [ai-agent-labo.com](https://ai-agent-labo.com/)** 。生きた実例として覗いてみてください。

**コスト・トークンの注意（公式）** ✅：小さいタスクに複数エージェントやループは不要（安く速いモデルで十分なことも多い）／成功・停止条件をはっきり決める／本格運用の前に小規模で試す／決まりきった作業はスクリプト化／必要以上の頻度でルーティンを回さない／`/usage` で使用量を確認。各ループは**隔離環境（worktree / クラウドボックス）**で回すと安全＆並列化できる ✅。

## 第5章 これからのSEO担当者の仕事

結論は、**「AIに記事を書かせる時代」から「AIが改善し続ける仕組みを設計する時代」へ**。ボトルネックは「執筆」から「オーケストレーション（ループ設計）」へ移る。

- **北極星を定める** — 会社のミッション・事業目標に根ざした長期ゴールを言語化し、エージェントと共有する。すると優秀なエージェントは新しい作業を主体的に提案しはじめる ✅。
- **検証責任は人間に残る** — ループの出力をレビューしないと理解が痩せる。人は「書く人」でも「丸投げする人」でもなく、**ループを設計し、停止条件を決め、要所で検証する監督**になる ✅。
- **段階的に信頼を広げる** — 最初は人が全部レビュー、実績が出たタスクから自律範囲を広げる。週次で「教訓と失敗」を記録させ、同じ失敗を繰り返さない ✅。

数字の面でも追い風はある：企業マーケの34%が自律エージェントを本番運用（2025Q4は14%）🔶、BCGはAIワークフローで工数25〜40%削減としている 🔶。ただし数値は二次情報なので「〜とされる」で扱う。

- AIを「ライター」ではなく「運用システム」として使う
- KPIは「書いた本数」から「回るループの数と質」へ
- まず一つ、いつも手でやっている繰り返し作業をループ化してみる（公式の“はじめの一歩”）✅
